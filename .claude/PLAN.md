## Key findings summary

**What is actually wired and running today:**
- iced app boots, login screen renders, settings + keychain persist
- `xmpp_subscription()` spawns `run_engine()`; engine connects via `tokio-xmpp`,
  sends roster-get + carbons-enable + presence on login
- Incoming roster, messages, and presence are dispatched to `ChatScreen`
- `ChatScreen` + `SidebarScreen` + `ConversationView` render and can send messages
- SQLite schema + migrations exist; `message_repo`, `conversation_repo`,
  `roster_repo` are implemented and tested — **none are called from the running app**
- Every XEP module (`stream_mgmt`, `mam`, `catchup`, `sync`, `avatar`,
  `muc`, `bookmarks`, `blocking`, `disco`, `message_mutations`, `file_upload`,
  `link_preview`, `console`, `command_palette`, `xmpp_uri`, `ignore`,
  `push_cleanup`, `account`, `conversation_sync`, `presence_machine`,
  `entity_time`) is implemented with full tests — **none are called by the engine**
- `OccupantPanel` UI widget exists but is never rendered
- `styling.rs` (XEP-0393 message styling) is implemented but not used in `ConversationView`
- `notifications.rs` is implemented but never called
- `connection/tcp.rs` and `connection/dns.rs` are empty stubs with `todo!()` — **not used** (engine uses `tokio-xmpp` directly, which is fine)
- `PresenceUpdated` event is received but not acted on in the UI (no online/offline indicator)
- `CatchupFinished` event variant exists but is only logged, never triggered
- `i18n` module declaration exists in `main.rs`/`lib.rs` but **the directory/file does not exist** (will cause compile failure if used)

**The rustls CryptoProvider panic:**
`tokio-xmpp` uses `rustls` with `ring` as the crypto backend. When multiple
`rustls` consumers are linked, one must call
`rustls::crypto::ring::default_provider().install_default()` once at startup
before the first connection attempt. Without it the `AsyncClient` constructor
panics with "no CryptoProvider". This must be fixed in `main.rs`.

## Immediate bugs (must fix first)

- [x] ✅ **B1: rustls CryptoProvider panic on connect** — `main.rs`
  Call `rustls::crypto::ring::default_provider().install_default().ok()` near
  the top of `fn main()`, before `iced::application(...)` runs. Without this
  the first connection attempt panics. The `.ok()` silences "already installed"
  errors so tests can also call it safely.
  **Done when**: connecting to a real XMPP server no longer panics on the first
  attempt; `cargo test` continues to pass.

- [x] ✅ **B2: `i18n` module declared but does not exist** — `main.rs`, `lib.rs`
  `pub mod i18n;` is declared in both files but no `src/i18n.rs` or
  `src/i18n/mod.rs` exists. This will panic with a compile error the moment
  anyone tries to `use crate::i18n`. Either create a minimal stub
  (`pub fn t(key: &str) -> &str { key }`) or remove the declaration until
  the feature is implemented.
  **Done when**: `cargo build` succeeds with no "file not found for module i18n"
  error.

- [x] ✅ **B3: Presence-availability changes are silently dropped** — `src/ui/mod.rs:204`
  `XmppEvent::PresenceUpdated` is matched and logged, but `chat.on_presence()`
  is never called, so the sidebar never shows online/offline status.
  **Done when**: `SidebarScreen` shows a visual indicator when a contact's
  presence changes (even a simple dot or "(away)" suffix is sufficient).
